## Problem Statement

Developers using Claude Code have no way to see and act on GitHub notifications from within their working session. They context-switch to email, GitHub web UI, or mobile to check PRs, issues, discussions, and review requests. When they see something that needs a response, they can't immediately use their AI coding agent to help. For open source maintainers, this is worse: they're bottlenecks on their own repos, manually triaging dozens of notifications per day.

breeze brings GitHub notifications into Claude Code, shows them in the statusline like a phone notification bar, and lets the agent summarize, suggest solutions, and take action on the user's behalf.

## What Makes This Cool

The notification inbox is the trojan horse. The real product is an autonomous GitHub presence that acts on your behalf. Existing tools (CodeRabbit, PR Agent, Hivemoot) are all cloud-hosted or GitHub Actions. None live inside your working session where the agent has your codebase context, conversation history, and intent.

The north star: agent handles 90% of your GitHub interactions autonomously. You just review the 10% that need human judgment. The v1 proves the interaction model. The agent evolution comes next.

## Constraints

- GitHub is the only message source (no Slack, email, Linear)
- `gh` CLI must be available and authenticated on the user's machine
- Local-first, no server, no webhooks
- Ships as a Claude Code skill at `agent-team-foundation/breeze-demo`
- Statusline is read-only display, all interaction via `/breeze` skill
- Agent summarizes notifications and suggests solutions (not just raw data)

## Premises

1. GitHub is the only message source for v1
2. `gh` CLI is always available and authenticated. breeze doesn't handle auth
3. Statusline is read-only notification display. All interaction through `/breeze` skill
4. A local background loop polls GitHub on an interval. Remote scheduled agent is a future evolution (Approach C), not v1
5. Agent summarizes context and suggests solutions when user picks a notification
6. Repo lives at `agent-team-foundation/breeze-demo`, ships as a Claude Code skill

## Approaches Considered

### Approach A: Minimal Skill
Single Claude Code skill that polls `gh api notifications` on demand when you type `/breeze`. Statusline updated by a hook on session start. No background process.
- Effort: S | Risk: Low
- Pros: Simplest possible, no moving parts
- Cons: Not real-time, notifications only refresh on session start or manual invoke

### Approach B: Skill + Cron Loop (CHOSEN)
`/breeze` skill for interaction + a cron/loop job that polls GitHub every 2-5 min and writes to a local inbox file (`~/.breeze/inbox.json`). Statusline reads the inbox file. Agent summarizes and suggests solutions when you pick a notification.
- Effort: M | Risk: Medium
- Pros: Near-real-time, statusline always current, inbox persists between sessions, natural stepping stone to full autonomy
- Cons: Background loop needs to be running, more moving parts than A

### Approach C: Skill + Cron + Remote Schedule
Everything in B, plus remote scheduled agent that runs when machine is off. Remote agent handles polling and can take simple auto-actions.
- Effort: L | Risk: High
- Pros: Works 24/7, closest to full autonomy vision
- Cons: Remote scheduling adds complexity, state sync between remote/local

## Recommended Approach

**Approach B: Skill + Cron Loop.** It's the sweet spot. Real-time notifications, local-first, and the cron loop is a natural stepping stone to C. Approach A is too limited to demo the vision. C adds remote complexity before the local UX is proven.

### Architecture

```
+-------------------+     writes      +---------------------+
|  Cron/Loop Job    |  -----------→   | ~/.breeze/inbox.json |
|  (every 2-5 min)  |                 +---------------------+
|  gh api notifs    |                        |
|  + enrich context |                        | reads
+-------------------+                        ↓
                                    +-------------------+
                                    | Statusline Hook   |
                                    | "breeze: 3 PRs"   |
                                    +-------------------+

User types /breeze
        ↓
+-------------------+     reads      +---------------------+
| /breeze Skill     |  -----------→  | ~/.breeze/inbox.json |
|                   |                +---------------------+
| 1. Show inbox     |
| 2. User picks one |     gh api     +---------------------+
| 3. Load context   |  -----------→  | GitHub API          |
| 4. Summarize      |                | (PR diff, thread,   |
| 5. Suggest action |                |  issue body, etc.)  |
| 6. Execute action |  -----------→  | gh pr review,       |
|    (with confirm) |                | gh issue comment... |
+-------------------+                +---------------------+
```

### Components

**1. Poller (background loop)**
- Implementation: a shell script installed as a system-level launchd plist (macOS) or crontab entry (Linux). This ensures the poller survives Claude Code session end. Claude Code `/loop` is session-scoped and would die when the user closes the terminal, so it is NOT suitable as the primary mechanism. The setup script creates and loads the launchd/crontab entry automatically
- Runs `gh api /notifications` with appropriate filters
- **Two-tier enrichment:** at poll time, fetches only lightweight metadata (notification id, type, repo, title, last actor, timestamp, reason). Full enrichment (PR diff, comment thread, issue body) happens on-demand when the user selects a notification in `/breeze`
- Writes structured JSON to `~/.breeze/inbox.json` using atomic writes (write to `~/.breeze/inbox.json.tmp` in the same directory, then `mv` to `inbox.json`) to prevent corruption from concurrent reads. Both files must be on the same filesystem for `mv` atomicity
- Priority ordering: review requests > mentions > assigned > participating > everything else
- Optionally marks notifications as read on GitHub after processing (default: off, per config `auto_mark_read: false`)

**2. Statusline Hook**
- Registers via Claude Code's statusline configuration (a shell command in `~/.claude/settings.json` under the `statusline` key that runs on an interval and displays output). If statusline integration is not available in the user's Claude Code version, the `/breeze` skill itself prints the notification count as a header when invoked
- Reads `~/.breeze/inbox.json` and outputs a one-line summary
- Displays: `breeze: 3 PRs . 1 issue . 2 mentions`
- Shows nothing if inbox is empty or file doesn't exist

**3. `/breeze` Skill**
- Reads inbox.json, renders numbered list with one-line summaries
- User picks a number or describes what they want to see
- Skill loads full context via `gh` CLI (PR diff, full comment thread, related code)
- Agent summarizes the situation and suggests an action
- User gives natural language instruction ("approve this", "comment asking for tests", "close as wontfix")
- Skill translates to `gh` CLI command, shows confirmation, executes

**4. Action Executor**
- Whitelist of safe actions: comment, review, approve, request changes, close, react, label
- No destructive actions without explicit confirmation (merge, delete branch, force push)
- Posts with footer: `sent via breeze on behalf of @username`

### inbox.json Schema

```json
{
  "last_poll": "2026-04-14T16:00:00Z",
  "notifications": [
    {
      "id": "github-notification-id",
      "type": "PullRequest",          // PullRequest | Issue | Discussion | Release | etc.
      "reason": "review_requested",   // review_requested | mention | assign | participating | etc.
      "repo": "owner/repo",
      "title": "Fix auth token refresh",
      "url": "https://github.com/owner/repo/pull/47",
      "number": 47,                  // nullable, only present for PullRequest and Issue types
      "last_actor": "username",
      "updated_at": "2026-04-14T15:55:00Z",
      "priority": 1,                  // 1=review_request, 2=mention, 3=assign, 4=participating, 5=other
      "unread": true
    }
  ]
}
```

Full enrichment (PR diff, comment thread, issue body) is fetched on-demand, not stored in inbox.json.

## Error Handling

- **`gh` not authenticated:** On first run, check `gh auth status`. If not authenticated, print "breeze requires `gh auth login` first" and exit. Do not attempt to handle auth.
- **Network failure / API errors:** If `gh api` returns non-zero, skip this poll cycle and retry next interval. Write nothing to inbox.json (preserve last good state).
- **Rate limiting:** Two-tier enrichment keeps poll-time API calls to ~1 per notification (just the notifications list endpoint + lightweight metadata). Full enrichment on-demand means rate usage scales with user interaction, not notification volume. If GitHub returns 429, back off for the duration specified in the `Retry-After` header.
- **File corruption:** Atomic writes (write to tmp, then mv) prevent partial reads. If inbox.json is somehow invalid, the skill falls back to a fresh `gh api /notifications` call.
- **Poller not running:** If inbox.json doesn't exist or is stale (>10 min old), `/breeze` skill shows a warning and offers to fetch notifications on-demand as a fallback.

## Notification Status System

Breeze tracks its own status per notification, independent of GitHub's read/unread.
This is the foundation for agent autonomy — agents need to know what's been handled,
what's in progress, and what needs human judgment.

### Five States

```
                    ┌─────────────────────────────────────────────┐
                    │                                             │
                    ▼                                             │
  [new notif] → ( open ) ──→ ( claimed ) ──→ ( resolved )        │
                  │  ▲          │    ▲           │                │
                  │  │          │    │           │ new activity   │
                  │  │  timeout │    │ extend    │ on GitHub      │
                  │  │  (5 min) │    │           └────────────────┘
                  │  │          ▼    │
                  │  └──────────┘    │
                  │                  │
                  │  ┌───────────────┘
                  │  │ new activity
                  │  │
                  ▼  │
              ( pending ) ── agent acted, waiting on someone else
                  │
                  │ new activity
                  ▼
               ( open )

              ( snoozed ) ── human said "not now"
                  │
                  │ timer expires OR new activity
                  ▼
               ( open )
```

| Status | Who sees it | Meaning | Example |
|--------|------------|---------|---------|
| **open** | Statusline, agents, humans | Needs action, no one's on it | New PR review request just arrived |
| **claimed** | Claimed agent only | Locked, someone is actively working | Agent is reading diff and writing review |
| **pending** | Hidden from statusline | Agent acted, waiting on external response | Requested changes, waiting for author to push |
| **snoozed** | Hidden from statusline | Human deferred it | "Snooze for 3 days" on a feature request |
| **resolved** | Hidden from statusline | Done, no action needed | PR approved, issue closed |

### Statusline only shows `open` count

```
/breeze: 12 PRs · 3 issues · 1 discussion
         ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
         only open notifications counted
```

### Claim Mechanism (Agent Lock)

Uses filesystem-based atomic locks to prevent two agents from handling the same notification:

```
~/.breeze/claims/<notification-id>/
    claimed_by    — agent session ID
    claimed_at    — ISO timestamp
    action        — what the agent is doing ("reviewing PR", "commenting on issue")
```

- **Claim:** `mkdir ~/.breeze/claims/<id>` (atomic on POSIX, fails if already exists)
- **Release:** `rm -rf ~/.breeze/claims/<id>` after action completes
- **Timeout:** If `claimed_at` is older than 5 minutes, any agent can reclaim (the previous agent crashed)
- **Extend:** Agent writes a new `claimed_at` if its action takes longer than expected

### Status Storage (`~/.breeze/status.json`)

```json
{
  "<notification-id>": {
    "status": "pending",
    "changed_at": "2026-04-14T16:05:00Z",
    "changed_by": "agent-session-abc123",
    "reason": "Requested changes on PR, waiting for author",
    "snooze_until": null,
    "github_updated_at": "2026-04-14T16:00:00Z"
  }
}
```

- Notifications NOT in status.json are implicitly `open`
- `github_updated_at` records what `updated_at` was when the status was set
- The poller compares this against the current GitHub `updated_at` to detect new activity
- Resolved entries are kept for 7 days (audit trail), then garbage collected

### Poller State Machine

On each poll cycle, the poller reads status.json and detects transitions:

1. **resolved + new activity** (GitHub `updated_at` > `github_updated_at`) → back to `open`
2. **pending + new activity** → back to `open` (the thing you were waiting for happened)
3. **snoozed + new activity** → back to `open` (new info overrides snooze)
4. **snoozed + timer expired** (`snooze_until` < now) → back to `open`
5. **claimed + timeout** (5 min) → back to `open` (agent crashed)
6. **claimed + new activity** → stay `claimed` (let the active agent handle it)

### Agent Autonomy UX

When an agent handles a notification, it follows a confidence-based escalation model:

```
Agent reads notification
    ↓
Agent assesses: can I handle this?
    ↓
┌─────────────────────────────────────────────────┐
│ HIGH confidence (>80%)                           │
│ e.g. docs-only PR, obvious duplicate issue       │
│                                                  │
│ Agent acts, then presents a REVIEW CARD:         │
│ ┌──────────────────────────────────────────┐    │
│ │ HANDLED: PR #301 "fix typo in README"     │    │
│ │ Action: Approved and merged               │    │
│ │ Confidence: 95%                           │    │
│ │ Reason: Docs-only, 1 line change, tests  │    │
│ │         pass, trusted contributor         │    │
│ │ [Undo] [View on GitHub]                   │    │
│ └──────────────────────────────────────────┘    │
│ → status: resolved                               │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ MEDIUM confidence (40-80%)                       │
│ e.g. code change PR, non-trivial issue           │
│                                                  │
│ Agent summarizes and SUGGESTS, waits for human:  │
│ ┌──────────────────────────────────────────┐    │
│ │ NEEDS REVIEW: PR #305 "add OAuth"         │    │
│ │ Summary: Adds OAuth2 login flow, 340      │    │
│ │          lines, touches auth module        │    │
│ │ Suggestion: Request changes — no tests    │    │
│ │             for token refresh edge case    │    │
│ │ Confidence: 60%                           │    │
│ │ [Approve suggestion] [Override] [Skip]    │    │
│ └──────────────────────────────────────────┘    │
│ → status: open (human decides)                   │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ LOW confidence (<40%)                            │
│ e.g. security issue, breaking change, conflict   │
│                                                  │
│ Agent flags for human with full context:          │
│ ┌──────────────────────────────────────────┐    │
│ │ ESCALATION: Issue #290 "Security vuln"    │    │
│ │ Why I can't handle this: Touches auth     │    │
│ │   module, potential breaking change,       │    │
│ │   needs domain knowledge I don't have      │    │
│ │ Context: [full summary loaded]             │    │
│ │ [Take over] [Snooze] [Assign to someone]  │    │
│ └──────────────────────────────────────────┘    │
│ → status: open (human must act)                  │
└─────────────────────────────────────────────────┘
```

### Review Summary

When the human opens `/breeze`, they see a dashboard:

```
/breeze inbox — 15 open · 3 claimed · 5 pending · 1 snoozed

AGENT ACTIVITY (last 24h):
  Resolved: 8 notifications
  Escalated: 2 (needed human judgment)
  Pending: 5 (waiting on others)

NEEDS YOUR ATTENTION (open):
  1. [PR] paperclip-tree #305 — add OAuth (suggestion: request changes) 60%
  2. [Issue] first-tree #90 — security vuln (escalation) 20%
  ...

RECENTLY HANDLED BY AGENT:
  ✓ PR #301 — fix typo in README (approved, 95%)
  ✓ PR #298 — update deps (approved, 90%)
  ✓ Issue #285 — duplicate of #42 (closed, 88%)
  [Undo any]
```

The human can review what the agent did, undo anything they disagree with, and handle
the escalations. This is the "10% human review" model — the agent does the work, the
human makes the judgment calls.

- User can see GitHub notification count in statusline without opening browser

- Poller runs reliably in the background without manual intervention
